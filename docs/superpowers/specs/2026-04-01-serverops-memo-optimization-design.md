# ServerOps 备忘录与 AI 助手优化设计

## 概述

**项目**：ServerOps 前端与交互优化  
**目标用户**：管理 20-50 台服务器的运维人员  
**核心目标**：
- 让 ServerOps 更像“服务器/服务备忘录”，方便查看和录入
- 增加一个 AI 对话框，作为主要检索入口之一
- AI 既能查系统内已录入的服务器/服务记录，也能补充相关说明和建议

**非目标**：
- 不实现 Agent
- 不执行运维动作
- 不做自动修改服务器记录
- 不引入外网搜索

---

## 需求分析

### 用户背景

- 用户以“备忘录”方式管理服务器和服务
- 当前规模约 20-50 台服务器，数量不大，但足以产生查找成本
- 用户既需要快速定位具体记录，也需要通过自然语言向 AI 询问“相关内容”

### 当前痛点

1. **找得到，但不够像备忘录**
   - 当前已有基础搜索，但只能按名称、IP、服务名做简单匹配
   - 用户仍然需要自己理解结果、进入详情、串联上下文

2. **录入成本偏高**
   - 添加服务器时仍需重复输入用户名、服务、备注等信息
   - 系统缺少便于 AI 理解的补充字段，例如标签、备注、别名

3. **AI 入口能力不足**
   - 现有 AI 组件更像普通聊天框，不是围绕“找服务器/服务/相关说明”设计
   - AI 无法稳定区分系统事实、补充知识和建议

### 目标能力

#### 目标 A：备忘录化查看

- 快速搜索服务器、服务、标签、备注
- 清楚展示“相关记录”
- 从 AI 回答可直接跳转到具体服务器或服务

#### 目标 B：备忘录化录入

- 录入服务器时支持标签、备注、服务备注、别名
- 降低重复填写成本
- 让录入内容更适合 AI 检索

#### 目标 C：AI 助手检索入口

- 用户可以直接提问：
  - “帮我找 AI 相关服务”
  - “哪台机器跑着 Ollama”
  - “数据库在哪些机器上”
  - “Open WebUI 一般怎么排查”
- AI 需要同时利用：
  - 系统内结构化资产数据
  - 本地补充知识库
- AI 以“助手”方式回答，但必须明确区分：
  - 哪些是系统记录
  - 哪些是补充说明
  - 哪些是建议或推断

---

## 现状评估

### 当前已具备

- 服务器列表页
- 服务列表页
- 基础搜索：名称 / IP / 服务名即时过滤
- 服务器录入与编辑
- AI 聊天组件

### 当前缺口

- 没有标签系统
- 没有服务器备注 / 服务备注 / 别名
- AI 没有围绕检索任务设计
- AI 回答无法结构化呈现相关记录
- 缺少本地知识库与资产检索的统一编排

---

## 设计原则

1. **AI 是入口，不是替代 UI**
   - 仍保留列表、详情、搜索、编辑等传统交互
   - AI 用于更自然地定位和解释信息

2. **系统事实优先**
   - 回答中优先返回系统内已录入的数据
   - 补充知识只作为解释与建议，不覆盖事实

3. **面向备忘录，而不是面向自动化**
   - 强化查找、录入、备注、链接能力
   - 不进入执行命令、操作服务器、自动修复等范围

4. **第一版不做 Agent**
   - 后端只做检索增强问答
   - 不做多步骤任务编排

---

## 方案设计

### 1. 备忘录数据增强

在现有服务器和服务模型基础上，增加最小但高价值的可检索字段。

#### 服务器新增字段

```typescript
interface ServerInfo {
  id: string;
  name: string;
  ip: string;
  username: string;
  password?: string;
  sshKey?: string;
  status: 'online' | 'offline' | 'maintenance';
  lastChecked: string;
  tags: Tag[];
  aliases: string[];
  notes?: string;
  services: Service[];
}
```

#### 服务新增字段

```typescript
interface Service {
  name: string;
  healthUrl?: string;
  status: 'online' | 'offline' | 'checking';
  category?: string;
  aliases: string[];
  notes?: string;
}
```

#### 标签结构

```typescript
interface Tag {
  id: string;
  name: string;
  color: string;
}
```

#### 设计意图

- `tags`：适合分类和快速筛选，例如生产、数据库、AI、监控
- `aliases`：解决自然语言表达不一致的问题
- `notes`：记录人类能看懂、AI 也能检索的说明
- `category`：用于服务级归类，但不等同于模板

### 2. 传统搜索与筛选增强

保留现有即时搜索，并增强为备忘录式检索。

#### 搜索范围

- 服务器名称
- IP 地址
- 服务名称
- 标签
- 服务器备注
- 服务备注
- 别名

#### 筛选方式

- 搜索框即时过滤
- 标签点击筛选
- 状态筛选
- 服务分类筛选

#### 交互要求

- 搜索无结果时给出明确提示
- AI 回答中的“加入筛选”动作可回填到列表页筛选条件
- 列表页与 AI 面板共享同一份过滤语义，但展示方式不同

### 3. 录入体验优化

录入不是为了模板化，而是为了让信息更完整、更可检索。

#### 表单新增字段

- 服务器标签
- 服务器别名
- 服务器备注
- 服务分类
- 服务别名
- 服务备注

#### 录入优化

- 复制上一条记录
- 复制当前编辑中的服务器作为新记录
- 常用用户名记忆
- 最近使用标签优先展示
- IP 格式即时校验

#### 设计意图

- 减少重复录入
- 让“我记得它大概是干什么的”这种模糊信息能被存下来
- 为 AI 检索提供可用上下文

### 4. AI 助手对话框

AI 对话框升级为“检索增强助手”，作为主入口之一。

#### 定位

- 不是普通闲聊窗口
- 主要任务是帮助用户找到系统记录，并补充相关说明

#### 支持的问题类型

- 找记录：帮我找和 AI 有关的服务
- 找位置：哪个服务跑在某台机器上
- 找状态：有哪些离线服务
- 找说明：Open WebUI 是什么
- 找建议：vLLM 一般怎么排查
- 混合问题：哪台机器跑着 Ollama，顺便告诉我常见检查点

#### 回答结构

AI 回答固定分为四段：

1. `结论`
2. `相关记录`
3. `补充说明`
4. `下一步操作`

#### 回答要求

- `相关记录` 必须来自系统内数据
- `补充说明` 来自本地知识库
- 如果出现模型推断，必须用明显文案标记为“建议”或“可能”

#### 回答中的快捷动作

- 打开服务器详情
- 复制 IP 或 URL
- 应用为页面筛选

### 5. 补充知识库

AI 不只查资产，还需要查本地知识内容。

#### 知识内容来源

- 本地 Markdown 文档
- 运维说明
- AI 服务说明
- 部署记录
- 排障笔记

#### 第一版组织方式

- 采用本地目录管理
- 文档为 Markdown 或纯文本
- 不要求第一版上复杂向量数据库

#### 建议目录

```text
docs/knowledge/
  ai-services/
  troubleshooting/
  runbooks/
  notes/
```

#### 检索范围

- 服务介绍
- 常见问题
- 运维建议
- 排障步骤

### 6. AI 检索流程

第一版采用“检索增强问答”，不做 Agent。

#### 请求处理流程

1. 前端发送用户问题
2. 后端判断问题意图：
   - 偏记录检索
   - 偏知识说明
   - 偏操作建议
   - 混合
3. 后端先检索资产数据
4. 后端再检索知识库内容
5. 将结果按固定结构整理给模型
6. 模型输出结构化回答
7. 前端渲染为回答卡片和相关记录列表

#### 为什么先查资产再查知识

- 资产记录是系统事实，应优先级更高
- 知识库用于解释和建议，不应盖过已有记录

### 7. 前端交互改动

#### AI 面板

- 输入框支持自然语言提问
- 展示回答卡片
- 展示“相关服务器/服务”
- 支持快捷操作按钮

#### 服务器列表页

- 增加标签筛选栏
- 增加更丰富的搜索占位提示
- 增加备注与标签展示

#### 详情页

- 增加标签、别名、备注
- 服务卡片增加分类、别名、备注

#### 添加/编辑弹窗

- 增加标签、备注、别名相关字段
- 服务录入增加分类与备注

## 架构设计

### 高层结构

```text
[Frontend]
  |- Server List / Detail / Modal
  |- AI Assistant Panel

[Backend API]
  |- Server CRUD
  |- Tag CRUD
  |- Assistant Query API

[Retrieval Layer]
  |- Asset Search
  |- Knowledge Search

[Storage]
  |- SQLModel / SQLite
  |- Local Markdown Knowledge Base

[LLM]
  |- Structured answer generation
```

### 关键决策

#### ADR-1：AI 采用“检索增强助手”，不采用 Agent

**决策**：第一版只做检索增强问答  
**原因**：
- 更贴合“备忘录”目标
- 范围可控
- 降低实现复杂度

**代价**：
- 不支持复杂多步骤任务
- 不支持自动执行动作

#### ADR-2：知识库使用本地文档，不上复杂检索基础设施

**决策**：第一版以本地 Markdown 为主  
**原因**：
- 规模较小，先验证价值
- 运维说明本来就适合文档化

**代价**：
- 检索精度和扩展性不如专门向量检索方案

#### ADR-3：回答结构固定化

**决策**：AI 回答按结论、相关记录、补充说明、下一步操作输出  
**原因**：
- 降低回答发散
- 便于前端结构化渲染
- 便于区分事实和建议

## 优先级

### P0 - 必须实现

1. 标签系统
2. 服务器 / 服务备注与别名字段
3. AI 问答接口
4. 资产检索 + 知识检索的后端编排
5. AI 面板结构化回答渲染
6. AI 回答中的相关记录跳转

### P1 - 重要

1. 复制上一条 / 复制现有服务器
2. 最近使用标签与用户名记忆
3. 服务分类字段与筛选
4. AI 回答中的“应用筛选”

### P2 - 优化

1. 搜索高亮
2. 快捷键聚焦
3. 简洁模式
4. 更丰富的知识文档组织

## 风险与缓解

### 风险 1：AI 回答把系统事实和建议混在一起

**缓解**：
- 固定回答结构
- 资产检索结果单独传入
- 前端分区域展示

### 风险 2：知识库内容质量不稳定

**缓解**：
- 明确知识目录结构
- 优先使用人工整理过的 Markdown
- 回答中保留来源区分

### 风险 3：录入不完整导致 AI 找不到

**缓解**：
- 新增标签、备注、别名字段
- 优化录入流程
- 引导用户补充关键描述

### 风险 4：第一版检索效果一般

**缓解**：
- 先覆盖高价值字段
- 先做关键词和简单语义结合
- 后续再评估是否需要更强检索方案

---

## 验收标准

1. 用户可以通过 AI 提问查找服务器、服务、标签和备注内容
2. AI 回答中可以明确列出系统内相关记录
3. AI 回答可以补充本地知识库中的说明和建议
4. AI 回答中可以直接跳转到对应服务器详情
5. 服务器和服务支持标签、备注、别名录入
6. 列表页支持按标签、备注、别名进行搜索和筛选
7. 第一版不包含 Agent 和自动执行动作
